# 项目全局规范

## 1. 规范定位

- 本文件是本项目唯一有效的全局规范文件。
- 后续代码、文档、说明和交付内容必须优先遵守本文件。

## 2. 技术栈与目录职责

- 后端必须使用 FastAPI，后端代码放在 `backend/app`。
- 前端必须使用 React + Vite，前端源码放在 `frontend/src`。
- 生产环境必须由后端托管前端构建产物；前端静态产物目录为 `frontend/dist`。
- 非必要情况下禁止引入 Docker 作为本项目运行前提。

## 3. AI 能力与接口约定

- 普通生图、图片编辑、对话式生图必须优先走 OpenAI 兼容的 `/v1/responses` 接口。
- 图片生成能力必须基于 `image_generation` 工具实现。
- `/v1/responses` 生图调用层必须优先使用 OpenAI SDK，不得重新维护裸 `httpx` 生图请求实现。
- 生图调用层必须支持对 `429/502/503/504/upstream_error/timeout` 等瞬时失败的统一重试，并支持 stream 失败后自动回退到非 stream 请求。
- 对话式改图仅在用户显式上传或选择参考图时允许进入编辑流程，禁止自动猜测历史图片作为编辑输入。
- 对话模式参考图总数必须限制为最多 3 张。
- 对话模式和分镜模式传给 AI planner 的上下文必须只包含显式文字消息与已生成图片对应的文字提示词，禁止依赖隐藏的上游 `previous_response_id` 链作为上下文。
- 用户选择参考图时，后端必须按当前会话过滤；发送给 AI planner 时必须将参考图图片与其对应提示词/说明成对传入，并明确标识二者关系。
- 不同会话或不同任务中选中的参考图禁止相互串用；前端切换会话时必须清理或过滤当前已选参考图。
- 同一会话内用户已选参考图必须保持稳定，禁止因发送对话或分镜消息自动取消；仅允许用户手动取消、切换会话、新建会话或删除会话时清理/过滤。
- 同一会话内候选参考图片列表必须随新生成图片增加而更新，但候选列表变化不得自动改变已选参考图。
- 对话模式和分镜模式必须支持 AI planner 与 image_generation 使用不同模型和不同 provider；planner 负责对话、意图判断和提示词撰写，image_generation 负责实际生图执行。
- AI planner 输出的生图提示词必须明确只对应一张图片；若参考图是用户本轮上传图片且没有历史生图提示词，禁止为该参考图编造对应提示词。
- 分镜连续生图必须先由 AI 规划镜头列表和人物/场景一致性概述，再按镜头顺序逐张生成；第 2 个及后续镜头必须以上一镜头输出图作为 edit 输入，禁止改成一次性合图或普通并发批量生图。
- 分镜连续生图的每张输出必须对应一个独立镜头，并保存中文镜头名与镜头顺序。
- 分镜连续生图失败、取消或意外中断时，历史记录必须保留已规划镜头、每个镜头状态、已完成图片和失败原因；历史详情必须支持基于已完成镜头继续重试。

## 4. 数据存储约定

- 本项目默认持久化存储必须使用 SQLite。
- 默认数据库文件位置必须为 `storage/app.db`，除非环境变量明确覆盖。
- 上传图片必须保存到 `storage/uploads`，生成结果必须保存到 `storage/outputs`。
- 提供商、应用设置、任务、会话、消息、提示词和图片记录必须优先落到 SQLite，不依赖浏览器本地存储作为唯一数据来源。

## 5. 运行与配置约定

- 默认服务端口必须为 `8010`，除非环境变量 `PORT` 明确覆盖。
- 与上游图片服务相关的默认配置必须通过 `.env` 或后端配置统一读取，禁止把密钥硬编码到后端逻辑中。
- 前端默认 provider、数据库中的默认 provider 与后端 `.env` 默认上游必须保持一致，禁止同时存在相互冲突的默认 `base_url`。
- 启动后端服务时必须优先使用项目内 `backend/.venv` 的 Python 解释器，禁止使用系统 Python、Anaconda 或其它项目外解释器直接启动本项目后端。
- 后端代码、后端依赖或 `.env` 配置发生变更后，若服务进程常驻运行，必须重启对应后端进程后再验证结果，禁止把未重启进程的运行结果视为最新代码行为。
- 任务执行必须支持后台运行；页面刷新或切换后不得丢失任务状态。

## 6. 生图参数约定

- 默认电脑壁纸生图参数必须为 `2560x1440` + `high`，禁止将 `1536x1024` 标注为电脑横屏 2K。
- `1536x1024` 仅允许作为 3:2 稳定横图选项展示。

## 7. 文案与交付约定

- 面向用户的默认产品文案、提示信息和错误说明应优先使用中文。
- 描述项目能力时，应基于“对话理解、生图执行、历史沉淀、任务可追踪”四类核心能力统一表述，避免与实际实现不一致的夸大描述。
