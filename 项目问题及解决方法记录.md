# 项目问题及解决方法记录

## 2026-05-02 运行中的后端进程未使用项目虚拟环境，且代码版本与常驻进程脱节

### 问题现象
- `8010` 端口上的实际后端进程使用的是系统 `Anaconda Python`，不是项目内 `backend/.venv`。
- `backend/app/main.py` 的磁盘修改时间晚于常驻后端进程启动时间，导致“代码已改，但运行结果仍是旧行为”。

### 影响
- 不同解释器中的依赖版本、模块行为和错误表现可能不一致。
- 排查数据库历史时看到的请求参数和当前磁盘代码不一致，容易误判为代码无效或逻辑混乱。

### 解决方法
- 启动或重启后端时统一使用：`backend\\.venv\\Scripts\\python.exe backend\\run.py`
- 每次修改后端代码、后端依赖或 `.env` 后，必须重启常驻后端进程，再做功能验证。
- 验证时先检查 `8010` 端口对应进程的解释器路径与启动时间，确认与当前代码版本一致。

## 2026-05-02 第三方网关在 `/v1/responses + image_generation` 链路上存在明显不稳定

### 问题现象
- 最近失败记录中同时出现了 `502 upstream_error`、`524 timeout`、`429 rate_limit_exceeded`。
- 文本 planner 请求往往成功，但真正进入 `image_generation` 阶段后更容易失败。

### 影响
- 项目表现为“有时能生图、有时突然失败”，用户会误以为前端或提示词本身有问题。
- 对话模式一次请求需要先走 planner、再走 image generation，本身比直接生图多一次 `/responses` 调用，更容易触发限流和网关超时。

### 解决方法
- 若需求已经明确，优先走直接 `generate` / `edit`，减少对话模式额外的一次 `/responses` 调用。
- 将并发控制在较低水平，并保留自动重试；若仍频繁出现 `524/502/429`，优先更换更稳定的上游 provider。
- 排查时优先区分“planner 成功、image_generation 失败”和“整个请求都失败”，不要把两者混为同一个问题。

## 2026-05-02 项目实现尚未真正对齐 `imagegen` skill 的核心容错逻辑

### 问题现象
- `imagegen` skill 使用 OpenAI SDK、流式优先提取最终图、流式失败回退非流式，并对 `429/502/503/504/upstream_error/timeout` 做统一重试。
- 当前项目自行实现了 `httpx + /responses` 调用层，默认重试次数更少，且稳定重试主要只覆盖网关超时场景。

### 影响
- 在相同上游下，skill 可恢复的瞬时错误，项目侧更容易直接暴露为失败。
- 项目与 skill 虽然都走 `/v1/responses`，但稳定性、报错质量和恢复能力并不等价。

### 解决方法
- 若项目目标是“把 skill 逻辑产品化”，优先复用或严格对齐 skill 的请求与容错策略，而不是重新实现一套更弱的调用层。
- 至少应统一以下能力：`base_url/api_key` 解析来源、流式最终图提取、流式失败回退非流式、对瞬时上游错误的统一重试策略。
- 在完成逻辑对齐前，不要把“项目已等价转换了 skill 能力”当作既定事实。

## 2026-05-02 Windows 下项目 venv 进程列表显示基础解释器路径

### 问题现象
- 使用 `backend\.venv\Scripts\python.exe backend\run.py` 启动后，Windows 进程列表中端口监听子进程可能显示为创建该 venv 的基础解释器路径。
- 本项目当前 venv 的基础解释器是 `D:\anaconda3\python.exe`，因此端口拥有者可能显示为 Anaconda Python。

### 影响
- 单看 `ExecutablePath` 容易误判服务没有使用项目虚拟环境。
- 若纯 Anaconda 环境缺少项目依赖，但服务仍能启动并导入项目新增依赖，说明服务实际使用的是 venv 的包环境。

### 解决方法
- 重启时仍以 `backend\.venv\Scripts\python.exe backend\run.py` 作为唯一启动命令。
- 验证时同时检查：启动父进程命令行、venv 中依赖是否存在、纯 Anaconda 中依赖是否不存在、服务是否能加载新增依赖后的接口。

## 2026-05-02 前端“电脑横屏 2K 高清”标签与实际像素不一致

### 问题现象
- 前端将“电脑横屏 2K 高清”配置为 `1536x1024`，该尺寸是 3:2，不是常见电脑壁纸 16:9。
- 数据库任务参数和实际输出图片都显示 `1536x1024`，说明前端和后端传参正常，但选项标签误导了用户。

### 影响
- 用户选择“电脑 2K 高清”后，生成结果会像 3:2 横图，而不是 `2560x1440` 电脑壁纸。
- 旧版保存的表单设置会继续带着 `1536x1024`，导致升级后仍可能沿用旧尺寸。

### 解决方法
- 将默认电脑壁纸尺寸统一为 `2560x1440` + `high`。
- 将 `1536x1024` 改名为“横屏 3:2 稳定”，禁止再标注为电脑横屏 2K。
- 对旧版未带 `settings_version` 的 `1536x1024` 保存设置做一次迁移，加载时自动切回新的默认电脑 2K 尺寸。

## 2026-05-02 分镜连续生图不能使用普通批量生图

### 问题现象
- 视频分镜首帧需要人物、服装、场景、光线和动作逻辑连续。
- 如果直接把多个镜头当作普通批量生图并发提交，每张图之间没有上一镜头画面作为约束，容易出现人物漂移、场景跳变和叙事断裂。

### 影响
- 用户会得到一组主题相近但不连续的图片，无法作为视频镜头首帧序列使用。
- 失败或中断时也难以判断已经完成到哪个镜头。

### 解决方法
- 分镜模式必须先由 AI 规划人物概述、场景概述和镜头列表。
- 第 1 镜头从文本或用户显式参考图生成；第 2 镜头开始必须把上一镜头输出图作为 edit 输入，再结合当前镜头提示词生成。
- 每个镜头生成后立即写入任务进度与镜头状态，失败时保留已完成镜头并标记失败镜头，便于续做和排查。

## 2026-05-02 对话参考图跨会话串台

### 问题现象
- 前端切换不同对话或任务后，已选参考图状态可能仍保留在当前表单里。
- 后端如果只按图片 id 查询参考图，不校验当前会话，会把其它对话中的图片传给当前 planner。

### 影响
- AI 会误以为当前需求引用了其它对话的图片，导致人物、场景或编辑目标串台。
- 分镜模式中参考图与提示词关系一旦混乱，会破坏后续镜头连续性。

### 解决方法
- 前端切换会话时必须清空或按当前 `conversation_id` 过滤已选参考图。
- 后端加载选中参考图时必须同时校验当前 `conversation_id`。
- 发送给 planner 时，参考图必须和对应提示词/说明成对排列，并用稳定 `ref` 标识二者关系。

## 2026-05-02 同一会话内参考图被发送消息自动清空

### 问题现象
- 对话或分镜消息提交成功后，前端会清空 `chatReferenceImages`。
- 用户明明想在同一会话内持续用同一张人物或场景参考图，下一轮对话却失去参考约束。

### 影响
- 连续改图和分镜首帧制作会出现人物、场景或风格约束突然消失。
- 新生成图片加入候选列表后，用户难以判断当前到底还选中了哪些参考图。

### 解决方法
- 同一会话内发送消息后只清空本轮上传文件，不自动清空已选历史参考图。
- 已选参考图只在用户手动取消、切换/新建/删除会话时清理或按当前会话过滤。
- 候选参考图片列表可以随新生成图片增加，但不能自动替换或取消用户已选参考图。

## 2026-05-02 Windows 本地无法直接执行 Bash 自检脚本

### 问题现象
- 在 Windows 本地执行 `bash scripts/self_check.sh` 时，`bash` 可能被解析到 WSL。
- 若本机未启用 WSL/Hyper-V，会在进入项目脚本前失败，并出现 `HCS_E_HYPERV_NOT_INSTALLED`。

### 影响
- 该失败只能说明当前 Windows Bash/WSL 环境不可用，不能直接判断 `scripts/self_check.sh` 或云服务器一键脚本有问题。
- 本地测试云服务器脚本时，容易把环境限制误判为项目脚本错误。

### 解决方法
- 云服务器或已安装 Git Bash/WSL 的环境中继续使用 `bash scripts/self_check.sh`。
- Windows 本地若未启用 WSL，应使用等价 PowerShell 检查：项目 venv 编译、`/api/health`、前端首页、`/api/settings`、`/api/providers`、`/api/app-settings`。
- 如需完整验证 Bash 脚本链路，先安装可用的 Git Bash 或启用 WSL/Hyper-V，再重新执行脚本。

## 2026-05-02 更新脚本不应覆盖云服务器业务数据库

### 问题现象
- 旧版 `update_project.sh` 在更新时会加载 `.env` 并调用 `configure_env.sh`。
- `configure_env.sh` 会初始化 SQLite，并更新第一条 provider 记录为 `.env` 中的默认上游配置。
- `start_background.sh` 也曾在检测到 `IMAGE_API_KEY` 环境变量时自动调用 `configure_env.sh`，导致更新后启动仍可能覆盖 provider。

### 影响
- 云服务器长期积累的 provider、settings、任务、会话、消息、提示词和图片记录存在被更新流程改写的风险。
- 用户在前端维护的 provider 配置可能被 `.env` 默认值覆盖，造成生图线路突然变化。

### 解决方法
- 更新项目时禁止调用会写业务数据的 `configure_env.sh`。
- 更新流程如需同步数据库变化，只能运行 schema 迁移脚本，执行创建缺失表、补充缺失列等向前兼容操作。
- `start_background.sh` 默认不得自动配置 `.env` 或 provider；只有显式设置 `AUTO_CONFIGURE_ENV=1` 时才允许执行配置初始化。

## 2026-05-03 分镜图片只在全部完成后刷新

### 问题现象
- 分镜模式按镜头顺序逐张生成图片，但旧逻辑主要在任务完成或轮询刷新时更新前端图片列表。
- 单个镜头图片保存后，后端没有单独推送“该镜头图片已保存”的事件，前端当前对话消息和任务详情也不会立即追加这张图。

### 影响
- 长分镜任务执行时，用户看不到每个镜头完成后的即时成果，容易误判任务卡住。
- 若后续镜头失败，用户需要刷新历史详情才更容易看到已完成镜头图片。

### 解决方法
- 每保存一张分镜镜头图片后，后端必须立即发布包含 task、shot、image 的任务级 SSE 事件。
- 前端收到事件后，必须立刻更新当前任务详情和当前对话助手消息的图片列表。
- 任务快照应携带已保存图片，保证历史详情、任务抽屉和 SSE 重连后状态一致。
