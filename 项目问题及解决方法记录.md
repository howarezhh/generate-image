# 项目问题及解决方法记录

## 2026-05-02 运行中的后端进程未使用项目虚拟环境，且代码版本与常驻进程脱节

### 问题现象
- `8010` 端口上的实际后端进程使用的是系统 `Anaconda Python`，不是项目内 `backend/.venv`。
- `backend/app/main.py` 的磁盘修改时间晚于常驻后端进程启动时间，导致“代码已改，但运行结果仍是旧行为”。

### 影响
- 不同解释器中的依赖版本、模块行为和错误表现可能不一致。
- 排查数据库历史时看到的请求参数和当前磁盘代码不一致，容易误判为代码无效或逻辑混乱。

### 解决方法
- 启动或重启后端时统一使用：`backend\\.venv\\Scripts\\python.exe backend\\run.py`
- 每次修改后端代码、后端依赖或 `.env` 后，必须重启常驻后端进程，再做功能验证。
- 验证时先检查 `8010` 端口对应进程的解释器路径与启动时间，确认与当前代码版本一致。

## 2026-05-02 第三方网关在 `/v1/responses + image_generation` 链路上存在明显不稳定

### 问题现象
- 最近失败记录中同时出现了 `502 upstream_error`、`524 timeout`、`429 rate_limit_exceeded`。
- 文本 planner 请求往往成功，但真正进入 `image_generation` 阶段后更容易失败。

### 影响
- 项目表现为“有时能生图、有时突然失败”，用户会误以为前端或提示词本身有问题。
- 对话模式一次请求需要先走 planner、再走 image generation，本身比直接生图多一次 `/responses` 调用，更容易触发限流和网关超时。

### 解决方法
- 若需求已经明确，优先走直接 `generate` / `edit`，减少对话模式额外的一次 `/responses` 调用。
- 将并发控制在较低水平，并保留自动重试；若仍频繁出现 `524/502/429`，优先更换更稳定的上游 provider。
- 排查时优先区分“planner 成功、image_generation 失败”和“整个请求都失败”，不要把两者混为同一个问题。

## 2026-05-02 项目实现尚未真正对齐 `imagegen` skill 的核心容错逻辑

### 问题现象
- `imagegen` skill 使用 OpenAI SDK、流式优先提取最终图、流式失败回退非流式，并对 `429/502/503/504/upstream_error/timeout` 做统一重试。
- 当前项目自行实现了 `httpx + /responses` 调用层，默认重试次数更少，且稳定重试主要只覆盖网关超时场景。

### 影响
- 在相同上游下，skill 可恢复的瞬时错误，项目侧更容易直接暴露为失败。
- 项目与 skill 虽然都走 `/v1/responses`，但稳定性、报错质量和恢复能力并不等价。

### 解决方法
- 若项目目标是“把 skill 逻辑产品化”，优先复用或严格对齐 skill 的请求与容错策略，而不是重新实现一套更弱的调用层。
- 至少应统一以下能力：`base_url/api_key` 解析来源、流式最终图提取、流式失败回退非流式、对瞬时上游错误的统一重试策略。
- 在完成逻辑对齐前，不要把“项目已等价转换了 skill 能力”当作既定事实。

## 2026-05-02 Windows 下项目 venv 进程列表显示基础解释器路径

### 问题现象
- 使用 `backend\.venv\Scripts\python.exe backend\run.py` 启动后，Windows 进程列表中端口监听子进程可能显示为创建该 venv 的基础解释器路径。
- 本项目当前 venv 的基础解释器是 `D:\anaconda3\python.exe`，因此端口拥有者可能显示为 Anaconda Python。

### 影响
- 单看 `ExecutablePath` 容易误判服务没有使用项目虚拟环境。
- 若纯 Anaconda 环境缺少项目依赖，但服务仍能启动并导入项目新增依赖，说明服务实际使用的是 venv 的包环境。

### 解决方法
- 重启时仍以 `backend\.venv\Scripts\python.exe backend\run.py` 作为唯一启动命令。
- 验证时同时检查：启动父进程命令行、venv 中依赖是否存在、纯 Anaconda 中依赖是否不存在、服务是否能加载新增依赖后的接口。

## 2026-05-02 前端“电脑横屏 2K 高清”标签与实际像素不一致

### 问题现象
- 前端将“电脑横屏 2K 高清”配置为 `1536x1024`，该尺寸是 3:2，不是常见电脑壁纸 16:9。
- 数据库任务参数和实际输出图片都显示 `1536x1024`，说明前端和后端传参正常，但选项标签误导了用户。

### 影响
- 用户选择“电脑 2K 高清”后，生成结果会像 3:2 横图，而不是 `2560x1440` 电脑壁纸。
- 旧版保存的表单设置会继续带着 `1536x1024`，导致升级后仍可能沿用旧尺寸。

### 解决方法
- 将默认电脑壁纸尺寸统一为 `2560x1440` + `high`。
- 将 `1536x1024` 改名为“横屏 3:2 稳定”，禁止再标注为电脑横屏 2K。
- 对旧版未带 `settings_version` 的 `1536x1024` 保存设置做一次迁移，加载时自动切回新的默认电脑 2K 尺寸。

## 2026-05-02 分镜连续生图不能使用普通批量生图

### 问题现象
- 视频分镜首帧需要人物、服装、场景、光线和动作逻辑连续。
- 如果直接把多个镜头当作普通批量生图并发提交，每张图之间没有上一镜头画面作为约束，容易出现人物漂移、场景跳变和叙事断裂。

### 影响
- 用户会得到一组主题相近但不连续的图片，无法作为视频镜头首帧序列使用。
- 失败或中断时也难以判断已经完成到哪个镜头。

### 解决方法
- 分镜模式必须先由 AI 规划人物概述、场景概述和镜头列表。
- 第 1 镜头从文本或用户显式参考图生成；第 2 镜头开始必须把上一镜头输出图作为 edit 输入，再结合当前镜头提示词生成。
- 每个镜头生成后立即写入任务进度与镜头状态，失败时保留已完成镜头并标记失败镜头，便于续做和排查。

## 2026-05-02 对话参考图跨会话串台

### 问题现象
- 前端切换不同对话或任务后，已选参考图状态可能仍保留在当前表单里。
- 后端如果只按图片 id 查询参考图，不校验当前会话，会把其它对话中的图片传给当前 planner。

### 影响
- AI 会误以为当前需求引用了其它对话的图片，导致人物、场景或编辑目标串台。
- 分镜模式中参考图与提示词关系一旦混乱，会破坏后续镜头连续性。

### 解决方法
- 前端切换会话时必须清空或按当前 `conversation_id` 过滤已选参考图。
- 后端加载选中参考图时必须同时校验当前 `conversation_id`。
- 发送给 planner 时，参考图必须和对应提示词/说明成对排列，并用稳定 `ref` 标识二者关系。

## 2026-05-02 同一会话内参考图被发送消息自动清空

### 问题现象
- 对话或分镜消息提交成功后，前端会清空 `chatReferenceImages`。
- 用户明明想在同一会话内持续用同一张人物或场景参考图，下一轮对话却失去参考约束。

### 影响
- 连续改图和分镜首帧制作会出现人物、场景或风格约束突然消失。
- 新生成图片加入候选列表后，用户难以判断当前到底还选中了哪些参考图。

### 解决方法
- 同一会话内发送消息后只清空本轮上传文件，不自动清空已选历史参考图。
- 已选参考图只在用户手动取消、切换/新建/删除会话时清理或按当前会话过滤。
- 候选参考图片列表可以随新生成图片增加，但不能自动替换或取消用户已选参考图。

## 2026-05-02 Windows 本地无法直接执行 Bash 自检脚本

### 问题现象
- 在 Windows 本地执行 `bash scripts/self_check.sh` 时，`bash` 可能被解析到 WSL。
- 若本机未启用 WSL/Hyper-V，会在进入项目脚本前失败，并出现 `HCS_E_HYPERV_NOT_INSTALLED`。

### 影响
- 该失败只能说明当前 Windows Bash/WSL 环境不可用，不能直接判断 `scripts/self_check.sh` 或云服务器一键脚本有问题。
- 本地测试云服务器脚本时，容易把环境限制误判为项目脚本错误。

### 解决方法
- 云服务器或已安装 Git Bash/WSL 的环境中继续使用 `bash scripts/self_check.sh`。
- Windows 本地若未启用 WSL，应使用等价 PowerShell 检查：项目 venv 编译、`/api/health`、前端首页、`/api/settings`、`/api/providers`、`/api/app-settings`。
- 如需完整验证 Bash 脚本链路，先安装可用的 Git Bash 或启用 WSL/Hyper-V，再重新执行脚本。

## 2026-05-02 更新脚本不应覆盖云服务器业务数据库

### 问题现象
- 旧版 `update_project.sh` 在更新时会加载 `.env` 并调用 `configure_env.sh`。
- `configure_env.sh` 会初始化 SQLite，并更新第一条 provider 记录为 `.env` 中的默认上游配置。
- `start_background.sh` 也曾在检测到 `IMAGE_API_KEY` 环境变量时自动调用 `configure_env.sh`，导致更新后启动仍可能覆盖 provider。

### 影响
- 云服务器长期积累的 provider、settings、任务、会话、消息、提示词和图片记录存在被更新流程改写的风险。
- 用户在前端维护的 provider 配置可能被 `.env` 默认值覆盖，造成生图线路突然变化。

### 解决方法
- 更新项目时禁止调用会写业务数据的 `configure_env.sh`。
- 更新流程如需同步数据库变化，只能运行 schema 迁移脚本，执行创建缺失表、补充缺失列等向前兼容操作。
- `start_background.sh` 默认不得自动配置 `.env` 或 provider；只有显式设置 `AUTO_CONFIGURE_ENV=1` 时才允许执行配置初始化。

## 2026-05-03 分镜图片只在全部完成后刷新

### 问题现象
- 分镜模式按镜头顺序逐张生成图片，但旧逻辑主要在任务完成或轮询刷新时更新前端图片列表。
- 单个镜头图片保存后，后端没有单独推送“该镜头图片已保存”的事件，前端当前对话消息和任务详情也不会立即追加这张图。

### 影响
- 长分镜任务执行时，用户看不到每个镜头完成后的即时成果，容易误判任务卡住。
- 若后续镜头失败，用户需要刷新历史详情才更容易看到已完成镜头图片。

### 解决方法
- 每保存一张分镜镜头图片后，后端必须立即发布包含 task、shot、image 的任务级 SSE 事件。
- 前端收到事件后，必须立刻更新当前任务详情和当前对话助手消息的图片列表。
- 任务快照应携带已保存图片，保证历史详情、任务抽屉和 SSE 重连后状态一致。

## 2026-05-03 分镜重试首镜头时丢失原始参考图

### 问题现象
- 分镜任务首次执行时，第 1 镜头可以使用用户上传图或已选历史图作为参考。
- 进入 `/api/tasks/{task_id}/retry` 后，重试逻辑只会回收已完成镜头的输出图作为 `previous_image`，不会重新装载原任务的输入参考图。
- 如果失败点就在第 1 镜头，或需要从没有已完成镜头的状态重新开始，重试会退化成纯文本 `generate`。

### 影响
- 首镜头重试结果会失去角色/场景锚点，人物外观和场景细节容易漂移。
- 用户会误以为“继续试一次”是在原参考条件下重跑，实际却已经换了约束条件。

### 解决方法
- 分镜重试必须同时保存并恢复原任务的输入参考图集合，而不只恢复已完成镜头输出图。
- 若第 1 镜头原本依赖参考图，重试第 1 镜头时必须继续走 `edit` 并把原始参考图重新传给 image_generation。
- 任务参数或独立表中应显式保存“首镜头参考图来源”和“后续镜头连续编辑来源”，避免重试时靠临时推断。

## 2026-05-03 分镜长链只依赖上一帧会逐步漂移

### 问题现象
- 分镜主链路在第 2 镜头及后续镜头中只把“上一镜头输出图”作为 edit 输入。
- 第 1 镜头使用过的角色设定图、场景参考图、道具参考图不会继续参与后续镜头执行。
- 文本里虽然保留了人物/场景概述，但图像级锚点已经只剩最近一帧。

### 影响
- 镜头数一长，人物脸型、服装细节、场景布局和光线风格会累计漂移。
- 一旦某一帧轻微跑偏，后续镜头会把偏差继续放大，难以作为稳定的视频首帧序列。

### 解决方法
- 后续镜头除上一帧外，还应允许持续带上固定角色锚点、场景锚点或关键道具锚点。
- 至少要支持“上一帧 + 固定角色参考图/设定图”的双层约束，而不是只保留单帧链式传递。
- 对长分镜应支持关键帧重锚定策略，例如每隔 N 个镜头重新注入角色设定图或用户指定参考图。

## 2026-05-03 分镜完成后把扩写版连续性提示词回灌进上下文，导致上下文膨胀

### 问题现象
- planner 先产出每个镜头的原始 prompt。
- 执行阶段又把“人物概述 + 场景概述 + 连续性要求 + 原始镜头 prompt”拼成更长的 `continuity_prompt`，并回写到 `shot.prompt`。
- 后续再次规划同一对话时，历史上下文会把这些扩写后的长 prompt 全量塞回 planner。

### 影响
- 对话越长、镜头越多，上下文会快速膨胀，增加 token 开销和 planner 输出不稳定性。
- planner 读到的是“执行期补强后的长提示词”，不再是原始分镜意图，容易造成信息重复、自我污染和规划跑偏。

### 解决方法
- 分离保存“planner 原始镜头 prompt”和“执行期 continuity_prompt”，禁止互相覆盖。
- 构造后续 planner 上下文时，优先回传原始镜头意图摘要，而不是执行期扩写全文。
- 对历史分镜上下文增加长度治理，例如只保留镜头摘要、关键连续性字段和用户确认过的不变量。

## 2026-05-03 分镜 SSE 断线或刷新后不会自动恢复实时事件流

### 问题现象
- 前端只在新建任务成功后启动 `EventSource`。
- 页面刷新、重新进入对话、或 SSE 连接临时报错后，前端没有为现存运行任务重新订阅事件流。
- 当前实现一旦触发 `onerror` 就直接关闭连接，后续主要依赖轮询补状态。

### 影响
- 分镜生成过程中的逐镜头完成通知会丢失实时性，用户只能靠轮询或手动刷新看到更新。
- 断线瞬间完成的镜头图片不一定能及时回灌到当前消息区域，体验会从“流式连续生成”退化成“间歇刷新”。

### 解决方法
- 页面初始化、切回正在执行的对话、或刷新后，必须自动为当前运行中的任务重建 SSE 订阅。
- `EventSource` 发生错误时不应直接永久关闭；应做指数退避重连，并在重连后用快照补齐当前状态。
- 任务级 SSE 快照除了任务状态外，还应能补发当前助手消息和已完成镜头图片，避免重连后界面缺图。

## 2026-05-03 继续改如果只保存历史图片 id，会在跨会话或非生成图场景下失效

### 问题现象
- 从历史记录或图库点击“继续改”时，前端若只把图片 id 填进已选参考图状态，而不把图片转成当前轮次的上传参考图，后端发送时仍会按当前会话过滤可选参考图。
- 生成模式、编辑模式、图库入口或模式切换后的继续改，常常不在原会话上下文里，导致图片虽然显示“已选”，实际提交时可能被后端过滤掉。
- 工作台底部若只显示“已指定 1/3 张”之类的文字，用户无法确认当前到底选中了哪一张图片。

### 影响
- 跨会话继续改会出现“前端看起来已选中，但真正发送时图片没带上”的隐性失败。
- 生成模式和图库里的继续改体验不稳定，用户很难判断当前是否真的把目标图带进了本轮任务。
- 只显示文字不显示缩略图，会放大误操作和排查成本。

### 解决方法
- 仅当目标图片属于当前会话同模式下可直接复用的生成图时，才允许用历史图片 id 作为参考图。
- 其它继续改场景必须把目标图片转换成当前轮次的上传参考图，确保提交时不依赖跨会话查询。
- 工作台底部必须显示已选参考图的真实缩略图、用途和移除入口，不能只显示数量或文字提示。

## 2026-05-03 只拦首页不拦接口和静态资源，会让密码页形同虚设

### 问题现象
- 项目若只在前端首页加一个密码弹窗，但不统一拦截 API、媒体资源和静态资源，用户仍可直接访问接口或资源地址。
- 前端单独做密码判断时，刷新页面、直接打开图片地址或访问 API 路径，都可能绕过页面层限制。

### 影响
- 访问控制只停留在界面层，无法真正保护项目内容和业务接口。
- 用户以为项目“加了密码”，但实际上接口、图片和前端资源仍可被直接访问，存在明显安全漏洞。

### 解决方法
- 访问控制必须统一放到后端入口层，对首页、静态资源、媒体资源和业务 API 做同一套登录态校验。
- 登录页应使用独立轻量 HTML 输出，避免在未登录时还依赖受保护的前端构建资源。
- API 未登录时返回统一未授权响应；页面访问未登录时跳转到密码页，确保所有入口行为一致。

## 2026-05-03 同一会话在任务运行中继续发消息会导致上下文串扰

### 问题现象
- 同一个 `conversation_id` 在已有 `queued` 或 `running` 任务时，前端仍可继续发送新消息。
- 对话模式与分镜模式也可能共用同一个会话，导致聊天与分镜历史混在一起。
- 从历史记录点击“继续对话”时，预期应该回到原会话继续，但切模式或再次发送后容易被错误地当成新会话或混入其它模式。

### 影响
- 同一会话在任务未完成时继续追加消息，会让任务状态、助手占位消息、参考图和上下文历史互相串扰。
- 聊天与分镜混用同一会话后，planner 读取到的历史语义不再单一，后续生成结果更容易跑偏。
- 历史记录里的“继续对话”会失去“继续原上下文”的意义，排查和追踪都会变得困难。

### 解决方法
- 对话模式与分镜模式必须使用各自独立的会话，不允许在同一 `conversation_id` 内混用。
- 同一会话存在 `queued` 或 `running` 任务时，前后端都必须阻止继续发送新消息；用户只能先停止该会话任务，或新建会话后继续。
- 从历史记录继续对话时，必须复用原 `conversation_id` 与原上下文；只有在用户主动切换到另一种模式时，才新建对应模式的新会话。

## 2026-05-03 图库如果按任务或时间碎片化分组，会打断同一会话的浏览连续性

### 问题现象
- 图库里的历史图片没有稳定按 `conversation_id` 聚合，导致同一会话生成的图片可能被拆到多个小分组里。
- 当同一会话图片较多时，列表会被整段撑长，用户很难快速扫到该会话的整体内容。
- 预览入口如果只绑定当前可见图片子集，上一张和下一张就无法覆盖同一会话的全部图片。

### 影响
- 用户难以把同一会话里的生成结果当作一个连续过程查看，尤其是对话连续改图或分镜任务。
- 图库页会因为单个会话图片过多而显得冗长，浏览效率下降。
- 预览弹窗里的前后切换范围不稳定，会出现“列表里明明还有图，但弹窗切不到”的割裂体验。

### 解决方法
- 图库分组必须优先按 `conversation_id` 聚合同一会话的全部历史图片；无会话归属的图片再按任务或单图兜底分组。
- 单个会话图片过多时，列表默认只显示部分预览缩略图，并提供“查看详情”展开全部图片。
- 图片预览时必须始终传入该会话的完整图片数组，而不是只传当前可见子集，确保上一张和下一张能覆盖同一会话全部图片。

## 2026-05-03 单全局并发限制会拖慢多提供商生图吞吐

### 问题现象
- 项目虽然支持配置多个 GPT 提供商，但最终生图执行仍按整个项目共用一个固定并发上限。
- 当某个 provider 已达到自身并发限制时，其它空闲 provider 也无法被优先利用。
- 用户在前端看不到当前任务究竟落到了哪个 provider，也无法判断池内还有多少 provider 空闲。

### 影响
- 多 provider 无法真正分摊生图压力，整体吞吐受最慢或最拥挤的单 provider 限制。
- 同时运行多个任务时，容易出现“还有 provider 空闲但总任务仍在排队”的体感问题。
- 排查限流、速度波动和 provider 稳定性时，缺少可见的 provider 使用信息，定位成本高。

### 解决方法
- 建立全局生图提供商池，普通生图、编辑、对话、分镜和分镜重试统一从池中选择最终执行 provider。
- 并发上限改为“单 provider 默认最多 3 个运行中任务”，项目总容量按“池内 provider 数 × 3”计算。
- 调度时先挑当前没有运行生图任务的 provider；如果所有 provider 都在运行任务，再选择运行中任务数最少的 provider。
- 任务记录、任务详情和前端提供商管理区都要展示当前实际使用的 provider 以及生图池总数、已使用数、空闲数。
- 前端在发起任务前必须先把最新 provider 池设置写回后端，避免刚调整池配置就提交任务时仍命中旧 provider 池。

## 2026-05-03 多参考图执行时缺少语义角色标注，容易引用错图

### 问题现象
- 分镜和对话模式最多可传 3 张参考图。
- 真正发给 image_generation 执行层时，每张图都只被标成通用的 `Input image N: primary reference image`。
- 执行层不知道哪张是角色设定、哪张是场景、哪张是服装或道具锚点。

### 影响
- 多参考图场景下，模型容易混淆各张图的职责，出现人物特征串用、场景误融合或风格污染。
- 分镜首镜头一旦引用错图，后续“以上一帧继续 edit”会把错误持续传递下去。

### 解决方法
- planner 或后端需要为每张参考图补充明确语义，例如“角色脸部锚点”“服装锚点”“场景锚点”。
- 执行层构造 input 时应把每张参考图的用途一起传给模型，而不是统一叫 primary reference image。
- 前端选择多参考图时也应允许用户标注用途，减少执行层猜测空间。
